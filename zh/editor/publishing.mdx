更改的发布方式取决于你当前所在的分支：

* **部署分支**：发布后会立即更新线上站点。
* **其他分支**：发布会创建一个拉取请求，便于你在部署到生产环境前先审阅更改。

<div id="pull-requests-and-reviewing-changes">
  ## Pull requests 和变更评审
</div>

Pull requests 允许你从分支发起变更，以便在合并到线上文档之前由他人进行评审。这有助于确保变更准确无误，并为团队提供协作完善内容的机会。

<Tip>
  即使是单人项目，pull requests 也有助于预览变更并维护清晰的更新历史。
</Tip>

<div id="creating-a-pull-request">
  ### 创建拉取请求
</div>

<Steps>
  <Step title="保存你的工作">
    确保已使用 **Save Changes** 将所有更改保存到你的分支。
  </Step>
  <Step title="创建拉取请求">
    在编辑器右上角选择 **Publish Pull Request**。
  </Step>
  <Step title="添加标题和描述">
    撰写清晰的标题和描述，说明：
    - 你做了哪些更改
    - 你为什么做这些更改
    - 需要重点审查的部分
  </Step>
  <Step title="创建并分享">
    选择 **Publish Pull Request**。编辑器会提供一个链接以查看你的拉取请求。

<div id="reviewing-pull-requests">
  ## 审查 pull request
</div>

创建 pull request 后：

1. **审查更改**：你和团队成员可以在 Git 提供商（如 GitHub 或 GitLab）中审查此 pull request。
2. **留下反馈**：添加评论或请求修改。
3. **进一步修改**：在网页编辑器中继续修改。保存后，编辑器会将更改推送到你的 pull request。
4. **批准**：当你对更改满意时，批准该 pull request。
5. **合并**：当你准备将更改部署到生产环境时，合并该 pull request。
