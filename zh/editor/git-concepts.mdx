Git 是一种版本控制系统，可用于跟踪文档的更改并与团队协作。借助 Git，你可以查看项目中每个文件的更改内容、时间以及原因；如有需要，还能轻松回退到文件的先前版本。

网页编辑器在后台会执行一些 Git 操作。理解这些概念将帮助你更高效地使用网页编辑器，并与在本地开发环境中工作的团队成员更好地协作。

<div id="core-git-concepts">
  ## 核心 Git 概念
</div>

<AccordionGroup>
  <Accordion title="Repository">
    你的文档源码存放处，保存所有文件及其历史。网页编辑器会连接到你的仓库以访问和修改内容。
  </Accordion>

  <Accordion title="Commit">
    在某个时间点保存的更改快照。每次提交都包含描述变更的消息，并在项目历史中留下永久记录。
  </Accordion>

  <Accordion title="Branch">
    分支指向仓库中的某个特定提交。你的在线文档由部署分支构建。你可以有任意数量的其他分支，其上的更改尚未发布到在线文档。如果想将某个分支的更改纳入在线文档，可通过 pull request 将该分支合并到部署分支。

    使用分支可以在不影响在线文档的前提下进行开发，安全试验新功能，并在发布前完成评审。
  </Accordion>

  <Accordion title="Deployment branch">
    项目的主要分支，在线文档内容由此构建。对此分支的更改会自动发布到你的文档站点。通常命名为 `main`，但你可以将任意分支设为部署分支。
  </Accordion>

  <Accordion title="Pull request">
    提议将分支上的更改合并进在线文档的方式。支持在更改上线前进行评审与讨论。通常称为 PR，在 GitLab 中也称为 merge request。
  </Accordion>

  <Accordion title="Diff">
    diff（差异）展示同一文件两个版本之间的变更。在审查 pull request 时，diff 会高亮显示新增、删除或修改的内容，便于识别具体变化。
  </Accordion>
</AccordionGroup>

<div id="how-the-web-editor-uses-git">
  ## 网页编辑器如何使用 Git
</div>

网页编辑器通过 [GitHub App](/zh/settings/github) 或 [GitLab 集成](/zh/settings/gitlab) 连接到你的 Git 仓库，并自动执行常见的 Git 操作。

当你：

* **打开文件**：编辑器会从仓库获取最新版本，确保你始终在处理最新内容。
* **进行更改**：编辑器将你的更改作为草稿跟踪，待你准备保存时可生成一次提交。
* **保存更改**：编辑器会基于你的更改创建一次提交，将你的工作保存在项目历史中。
* **创建分支**：编辑器会在仓库中创建一个新分支，任何有仓库访问权限的人都可以使用该分支进行协作和审阅。
* **在部署分支上发布**：编辑器会直接向部署分支提交并推送，从而立即发布你的更改。
* **在其他分支上发布**：编辑器会创建一个拉取请求，便于你在将更改合并到部署分支之前先获取他人的反馈。

<div id="git-best-practices">
  ## Git 最佳实践
</div>

每个团队都会形成自己的工作流程和偏好，但以下是一些通用的最佳实践，助你快速上手。

* **撰写清晰的提交信息**：使用主动语态，具体说明更改内容。
* **使用具有描述性的分支命名**：分支名应能说明所做工作，对查看你仓库分支的人一目了然。
* **保持分支单一职责**：让该分支的更改专注于某个特定任务或项目。
* **合并后删除分支**：在不再需要时删除分支，保持仓库整洁。
