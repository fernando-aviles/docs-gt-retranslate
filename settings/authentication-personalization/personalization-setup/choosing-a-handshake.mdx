<Info>
  This guide covers **Personalization** Handshake methods. For authentication, see [Authentication Handshake methods](/settings/authentication-personalization/authentication-setup/choosing-a-handshake).
</Info>

Users need to be authenticated to access personalized content. Choose from three methods:

1. **Shared Session**: Use your dashboard's session token
2. **JWT**: Pass user info via JWT in the URL
3. **OAuth 2.0**: Use PKCE flow with your OAuth server

## Prerequisites

<Tabs>
  <Tab title="Shared Session">
    - Dashboard hosted at your domain
    - Session stored in cookies
    - API endpoint at same origin/subdomain as dashboard
    - Docs hosted at same domain as dashboard

    Example domains:
    - Dashboard at `foo.com`: API must be `foo.com` or `*.foo.com`
    - Dashboard at `dash.foo.com`: API must be `dash.foo.com` or `*.dash.foo.com`
  </Tab>
  <Tab title="JWT">
    - Existing login flow
    - Ability to create JWT and redirect to docs
  </Tab>
  <Tab title="OAuth 2.0">
    - OAuth server with PKCE support
    - API endpoint accessible via OAuth token
  </Tab>
</Tabs>

## Comparison

<Tabs>
  <Tab title="Shared Session">
    ✅ Auto-login if user is in dashboard
    ✅ Persistent sessions
    ✅ Simple setup
    
    ❌ Requires backend request
    ❌ Needs dashboard with session auth
    ❌ Requires CORS setup
  </Tab>
  <Tab title="JWT">
    ✅ Lower API abuse risk
    ✅ No CORS needed
    ✅ Flexible API URLs
    
    ❌ Must modify login flow
    ❌ Separate login for dashboard and docs
    ❌ Requires re-login for data updates
  </Tab>
  <Tab title="OAuth 2.0">
    ✅ High security
    ✅ Flexible API URLs
    
    ❌ Complex OAuth setup
    ❌ Separate login for dashboard and docs
    ❌ May be excessive for simple needs
  </Tab>
</Tabs>
